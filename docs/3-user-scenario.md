## 2. 페르소나별 주요 사용 맥락

### 페르소나 A - 김지수 (21세, 대학생)

김지수는 4개 과목을 수강하며 매주 과제, 시험, 발표 일정이 겹치는 바쁜 대학생이다. 지금까지 카카오톡 메모나 수첩에 과제를 적어왔는데, 마감일이 지나도록 확인하지 못해 점수를 잃는 경험을 반복해왔다.

**주요 사용 흐름**: 김지수는 서비스에 가입한 뒤 그날 들은 수업에서 나온 과제를 바로 등록한다. 이후 매일 아침 앱을 열어 기한이 지난 과제가 빨간색으로 상단에 표시되는지 확인한다. 과제를 제출하면 즉시 완료 처리해 목록을 정리하고, 수업 계획이 바뀌면 마감일을 수정한다.

**주요 시나리오**: SC-01, SC-02, SC-03, SC-07

---

### 페르소나 B - 박준혁 (26세, 대학원생)

박준혁은 논문 준비와 인턴 지원을 병행하는 대학원생이다. 처리할 일이 많고 성격도 다양해서 완료한 일과 남은 일이 뒤섞여 보이면 우선순위 파악이 힘들다는 불편함을 느껴왔다. 완료 기록은 지우지 않고 남겨두되, 미완료 항목에 집중해서 보고 싶어한다.

**주요 사용 흐름**: 박준혁은 논문 관련 할일과 인턴 지원 할일을 각각 등록하고 마감일을 설정한다. 일이 완료되면 삭제하지 않고 완료 처리만 해서 기록으로 남긴다. 집중이 필요할 때는 상태 필터를 미완료로 설정해 남은 작업만 확인한다. 특정 작업을 빠르게 찾을 때는 키워드 검색을 활용한다.

**주요 시나리오**: SC-04, SC-05, SC-03

## 3. 핵심 시나리오 목록

### SC-01: 신규 회원가입 후 첫 할일 등록

**페르소나**: 김지수 (페르소나 A)
**사전 조건**: 서비스에 가입된 계정이 없음. 브라우저에서 my_todolist 주소에 처음 접속한 상태
**목표**: 회원가입을 완료하고, 오늘 수업에서 나온 첫 번째 과제를 등록한다

**시나리오 흐름**:
1. 로그인 화면(`/login`)에 진입한다. 하단의 "회원가입" 링크를 클릭한다.
2. 회원가입 화면(`/signup`)에서 이름(예: 김지수), 이메일(예: jisu@university.ac.kr), 비밀번호(예: pass1234)를 입력한다.
3. "가입하기" 버튼을 클릭한다.
4. 가입이 완료되고 자동으로 로그인 처리되어 할일 목록 화면(`/`)으로 이동한다.
5. 할일이 없어 목록이 비어있는 화면을 확인한다.
6. "할일 추가" 버튼을 클릭해 생성 폼(모달 또는 `/todos/new`)을 연다.
7. 제목에 "운영체제 3장 요약 제출", 마감일에 이번 주 금요일 날짜를 입력하고, 설명은 비워둔다.
8. "저장" 버튼을 클릭한다.
9. 할일 목록에 방금 등록한 과제가 즉시 나타나는 것을 확인한다.

**예상 결과**: 회원가입 완료 후 자동 로그인되고, 첫 번째 할일이 목록에 표시된다
**관련 US**: US-01, US-04
**관련 AC**: AC-01, AC-06

### SC-02: 로그인 후 마감 기한 경과 할일 확인 및 처리

**페르소나**: 김지수 (페르소나 A)
**사전 조건**: 가입된 계정이 있음. 마감일이 이미 지났으나 아직 완료 처리하지 않은 할일이 1개 이상 존재함
**목표**: 기한이 지난 과제를 확인하고 완료 또는 삭제 처리해 목록을 정리한다

**시나리오 흐름**:
1. 로그인 화면(`/login`)에서 이메일과 비밀번호를 입력하고 "로그인" 버튼을 클릭한다.
2. 할일 목록 화면(`/`)으로 이동한다.
3. 목록 최상단에 빨간색 테두리와 경고 아이콘이 표시된 "기한 초과" 그룹을 확인한다.
4. 해당 그룹 안에 "데이터베이스 과제 제출"이라는 항목이 있는 것을 발견한다.
5. 이미 제출한 과제임을 기억하고 항목 왼쪽 체크박스를 클릭해 완료 처리한다.
6. 해당 항목이 "기한 초과" 그룹에서 사라지고 완료 상태로 이동하는 것을 확인한다.
7. 기한이 지났고 더 이상 필요 없는 다른 항목은 삭제 버튼을 클릭한다.
8. 삭제 확인 다이얼로그에서 "삭제" 버튼을 클릭해 항목을 영구 삭제한다.

**예상 결과**: 기한 초과 항목이 시각적으로 구분되어 즉시 식별 가능하고, 완료 처리 및 삭제 후 목록이 정리된다
**관련 US**: US-02, US-05, US-07, US-08
**관련 AC**: AC-03, AC-10, AC-11, AC-09

### SC-03: 할일 수정 및 완료 처리

**페르소나**: 박준혁 (페르소나 B)
**사전 조건**: 로그인 상태. "논문 1장 초안 작성" 할일이 목록에 존재하며 마감일이 다음 주로 설정되어 있음
**목표**: 지도교수 피드백으로 마감일이 변경된 할일을 수정하고, 완료 후 완료 처리한다

**시나리오 흐름**:
1. 할일 목록 화면(`/`)에서 "논문 1장 초안 작성" 항목을 찾는다.
2. 해당 항목의 수정 버튼(연필 아이콘 또는 "수정" 버튼)을 클릭한다.
3. 수정 폼(모달 또는 `/todos/:id/edit`)이 열리며 기존 값(제목, 설명, 마감일)이 미리 채워진 상태로 표시된다.
4. 마감일을 이번 주 목요일로 앞당겨 변경한다.
5. "저장" 버튼을 클릭한다.
6. 목록에서 해당 할일의 마감일이 즉시 변경된 것을 확인한다.
7. 마감일이 앞당겨져 정렬 순서도 바뀐 것을 확인한다.
8. 초안 작성을 완료한 후 해당 항목의 체크박스를 클릭해 완료 처리한다.
9. 항목이 완료 상태로 변경되고 미완료 그룹에서 분리되는 것을 확인한다.
10. 기록으로 남기기 위해 삭제하지 않고 완료 상태로 유지한다.

**예상 결과**: 마감일 수정이 목록에 즉시 반영되고, 완료 처리 후 항목이 완료 상태로 분류되어 기록에 남는다
**관련 US**: US-06, US-07
**관련 AC**: AC-08, AC-10

### SC-04: 키워드 검색으로 특정 할일 찾기

**페르소나**: 박준혁 (페르소나 B)
**사전 조건**: 로그인 상태. 논문, 인턴 지원, 수업 관련 할일이 20개 이상 등록되어 있음
**목표**: 인턴 지원 관련 할일 중 "자기소개서"가 포함된 항목을 빠르게 찾는다

**시나리오 흐름**:
1. 할일 목록 화면(`/`)에서 상단 검색바를 클릭한다.
2. 검색어 "자기소개서"를 입력한다.
3. 입력과 동시에 또는 검색 버튼 클릭 후 목록이 갱신된다.
4. 제목 또는 설명에 "자기소개서"가 포함된 할일만 목록에 표시된다.
5. "카카오 인턴 자기소개서 작성" 항목을 찾아 마감일과 설명을 확인한다.
6. 검색어를 지우면 전체 할일 목록으로 돌아온다.

**예상 결과**: "자기소개서" 키워드가 제목 또는 설명에 포함된 할일만 필터링되어 표시된다. 검색 결과가 없으면 빈 목록이 표시된다(오류 메시지 없음)
**관련 US**: US-10
**관련 AC**: AC-06 (생성된 데이터 기준)

### SC-05: 상태 필터로 미완료 항목만 조회

**페르소나**: 박준혁 (페르소나 B)
**사전 조건**: 로그인 상태. 완료된 할일 10개, 미완료 할일 8개가 섞여 목록에 표시되어 있음
**목표**: 완료된 항목은 숨기고 아직 처리하지 못한 미완료 항목만 집중해서 확인한다

**시나리오 흐름**:
1. 할일 목록 화면(`/`)에서 완료된 항목과 미완료 항목이 섞여 표시되는 것을 확인한다.
2. 상태 필터 탭에서 "미완료(pending)" 항목을 클릭한다.
3. 목록이 즉시 갱신되어 미완료 상태의 할일 8개만 표시된다.
4. 완료된 항목은 목록에서 보이지 않는 것을 확인한다.
5. 미완료 항목 중 오늘 처리해야 할 것을 골라 작업 순서를 결정한다.
6. 다시 "전체" 탭을 클릭하면 완료/미완료 항목이 모두 다시 표시된다.

**예상 결과**: 미완료 필터 선택 시 pending 상태 할일만 즉시 표시되고, 완료 항목은 숨겨진다
**관련 US**: US-09
**관련 AC**: AC-10 (상태 구분 기준)

### SC-06: 미인증 접근 시 로그인 리다이렉트

**페르소나**: 김지수 (페르소나 A) - 미로그인 상태
**사전 조건**: 브라우저에 로그인 상태가 없음(토큰 없음 또는 만료). 이전에 방문한 기록으로 북마크에 할일 목록 URL(`/`)이 저장되어 있음
**목표**: (시스템 관점) 인증되지 않은 사용자가 보호된 페이지에 접근하지 못하도록 차단하고 로그인 페이지로 안내한다

**시나리오 흐름**:
1. 김지수가 북마크에 저장된 할일 목록 URL(`/`)에 직접 접근을 시도한다.
2. 브라우저에 저장된 토큰이 없거나 만료된 상태라 인증에 실패한다.
3. 시스템이 자동으로 로그인 화면(`/login`)으로 리다이렉트한다.
4. 김지수는 이메일과 비밀번호를 입력하고 로그인한다.
5. 로그인 성공 후 할일 목록 화면(`/`)으로 이동한다.

**예상 결과**: 미인증 상태에서 보호된 경로 접근 시 로그인 페이지로 자동 이동하며, 로그인 성공 후 할일 목록에 정상 접근한다
**관련 US**: US-02
**관련 AC**: AC-05, AC-07

### SC-07: 모바일에서 할일 등록 및 완료 처리

**페르소나**: 김지수 (페르소나 A)
**사전 조건**: 로그인 상태. 스마트폰 브라우저(화면 너비 375px 이상)로 접속 중
**목표**: 강의실에서 수업을 들으며 스마트폰으로 새로운 과제를 빠르게 등록하고, 이전에 등록한 과제를 완료 처리한다

**시나리오 흐름**:
1. 스마트폰 브라우저에서 my_todolist에 접속한다. 모바일 레이아웃으로 화면이 정상 표시된다.
2. 기존 할일 목록이 세로 스크롤 형태로 표시되는 것을 확인한다.
3. 하단 또는 상단의 "할일 추가" 버튼을 손가락으로 탭한다.
4. 생성 폼이 모바일 화면에 맞게 표시된다. 제목("알고리즘 5번 문제 풀기")과 마감일을 입력한다.
5. "저장" 버튼을 탭한다.
6. 목록으로 돌아와 새로 등록된 할일이 즉시 표시되는 것을 확인한다.
7. 이미 완료한 이전 과제 항목의 체크박스를 탭해 완료 처리한다.
8. 항목 상태가 완료로 변경되는 것을 확인한다.

**예상 결과**: 모바일 화면(375px 이상)에서 레이아웃 깨짐 없이 할일 등록 및 완료 처리가 정상 동작한다
**관련 US**: US-04, US-07
**관련 AC**: AC-06, AC-10, AC-13

## 4. 예외 및 오류 시나리오

### ERR-01: 중복 이메일로 회원가입 시도

**페르소나**: 신규 방문자 (이미 가입한 이메일 사용)
**사전 조건**: `jisu@university.ac.kr` 이메일로 이미 가입된 계정이 존재함
**목표**: (테스트 관점) 중복 이메일 가입 차단 여부를 확인한다

**시나리오 흐름**:
1. 회원가입 화면(`/signup`)에서 이름, 이메일(`jisu@university.ac.kr`), 비밀번호를 입력한다.
2. "가입하기" 버튼을 클릭한다.
3. 서버가 중복 이메일을 감지하고 400 오류를 반환한다.
4. 화면에 "이미 사용 중인 이메일입니다." 오류 메시지가 이메일 입력 필드 아래에 인라인으로 표시된다.
5. 계정이 생성되지 않고 회원가입 화면에 그대로 머무른다.

**예상 결과**: 중복 이메일 가입이 차단되고 사용자에게 명확한 안내 메시지가 표시된다
**관련 AC**: AC-02

---

### ERR-02: 잘못된 비밀번호로 로그인 시도

**페르소나**: 김지수 (페르소나 A)
**사전 조건**: 가입된 계정이 존재함. 비밀번호를 잘못 기억하고 있음
**목표**: (테스트 관점) 잘못된 자격증명으로 로그인 차단 여부를 확인한다

**시나리오 흐름**:
1. 로그인 화면(`/login`)에서 이메일과 잘못된 비밀번호(예: wrongpass)를 입력한다.
2. "로그인" 버튼을 클릭한다.
3. 서버가 자격증명 불일치를 확인하고 401 오류를 반환한다.
4. 화면에 "이메일 또는 비밀번호가 올바르지 않습니다." 오류 메시지가 표시된다.
5. 토큰이 발급되지 않고 로그인 화면에 그대로 머무른다.

**예상 결과**: 잘못된 비밀번호로 로그인이 차단되고 오류 메시지가 표시된다. 토큰이 발급되지 않는다
**관련 AC**: AC-04

---

### ERR-03: 필수 입력값 누락 후 할일 생성 시도

**페르소나**: 김지수 (페르소나 A)
**사전 조건**: 로그인 상태. 할일 생성 폼이 열려있음
**목표**: (테스트 관점) 필수 입력값 없이 저장 시도 시 유효성 검증 동작을 확인한다

**시나리오 흐름**:
1. 할일 생성 폼에서 제목을 입력하지 않고 마감일만 입력한 상태로 "저장" 버튼을 클릭한다.
2. 클라이언트 측 유효성 검증이 실행되어 제목 입력 필드 아래에 "제목은 필수 입력 항목입니다." 메시지가 인라인으로 표시된다.
3. API 요청이 발생하지 않고 폼에 그대로 머무른다.
4. 이번에는 제목은 입력했지만 마감일을 입력하지 않은 상태로 "저장" 버튼을 클릭한다.
5. 마감일 입력 필드 아래에 "마감일은 필수 입력 항목입니다." 메시지가 표시된다.
6. 제목과 마감일을 모두 입력한 뒤 저장하면 할일이 정상 생성된다.

**예상 결과**: 필수 입력값 누락 시 API 호출 없이 클라이언트에서 즉시 오류 메시지를 표시하고 생성이 차단된다
**관련 US**: US-04
**관련 AC**: AC-06

### ERR-04: 타인의 할일 접근 시도

**페르소나**: 박준혁 (페르소나 B) - 악의적 접근 가정
**사전 조건**: 박준혁이 자신의 계정으로 로그인되어 있음. 다른 사용자(김지수)의 할일 ID를 알고 있다고 가정
**목표**: (테스트 관점) 인증된 사용자라도 타인의 데이터에 접근하지 못하도록 차단되는지 확인한다

**시나리오 흐름**:
1. 박준혁이 자신의 Access Token을 사용해 김지수 소유의 할일 ID로 조회 API를 직접 호출한다 (예: `GET /todos/{김지수의_할일_ID}`).
2. 서버가 요청한 할일의 소유자와 토큰의 사용자를 비교한다.
3. 소유자가 일치하지 않아 403 오류를 반환한다.
4. 수정(`PUT /todos/{ID}`) 및 삭제(`DELETE /todos/{ID}`) 시도 시에도 동일하게 403 오류가 반환된다.
5. 박준혁의 할일 목록 조회(`GET /todos`)에는 자신의 할일만 반환되며, 김지수의 할일은 포함되지 않는다.

**예상 결과**: 인증된 사용자라도 타인의 할일에 대한 조회, 수정, 삭제가 403 오류로 차단된다
**관련 AC**: AC-12
