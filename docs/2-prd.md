## 1. 제품 개요

### 비전

학생이 복잡한 학사 일정과 과제를 놓치지 않도록 돕는, 단순하고 빠른 개인 할일 관리 도구를 만든다.

### 목적

기존 메모앱이나 노트 앱은 마감일 관리와 완료 추적에 최적화되어 있지 않다. my_todolist는 마감일 중심의 할일 목록과 기한 초과 시각화로 학생의 일정 누락을 방지한다.

### 제품 목표

| 목표 | 세부 내용 |
|------|---------|
| 학생 일정 누락 방지 | 마감 기한 경과 항목을 즉시 식별할 수 있도록 시각 구분 제공 |
| 빠른 할일 등록/완료 | 3단계 이내 할일 생성, 1회 클릭 완료 처리 |
| 안전한 개인 데이터 | JWT 인증으로 타인의 할일 접근 차단 |
| 신속한 MVP 출시 | 3일 이내 핵심 기능 구현 (2026-02-14 금요일 오후 목표) |

| 제외 기능 | 제외 이유 |

## 6. UI/UX 요구사항

### 주요 화면 목록

| 화면 | 경로 | 접근 조건 | 핵심 요소 |
|------|------|---------|---------|
| 로그인 | `/login` | 비인증 | 이메일/비밀번호 입력, 회원가입 링크 |
| 회원가입 | `/signup` | 비인증 | 이름/이메일/비밀번호 입력, 유효성 안내 |
| 할일 목록 | `/` (홈) | 인증 | 기한 초과 그룹, 할일 카드, 검색바, 필터, 정렬 |
| 할일 생성 | 모달 or `/todos/new` | 인증 | 제목/설명/마감일 입력 폼 |
| 할일 수정 | 모달 or `/todos/:id/edit` | 인증 (소유자) | 기존 값 pre-fill, 수정 후 저장 |

### 화면 흐름

```
[비인증 진입]
    │
    ├─ /login ──────── 로그인 성공 ──────── /  (할일 목록)
    │                                          │
    └─ /signup ─────── 가입 성공 ─ 자동 로그인 ┘

[할일 목록 화면 - /]
    │
    ├─ 기한 초과 그룹 (빨간 테두리/아이콘)
    │     └─ 완료 처리 / 수정 / 삭제
    │
    ├─ 일반 할일 그룹 (due_date 오름차순)
    │     └─ 완료 처리 / 수정 / 삭제
    │
    ├─ 상태 필터 탭 (전체 / 미완료 / 완료)
    ├─ 정렬 드롭다운
    ├─ 키워드 검색바
    └─ 할일 추가 버튼 ─── 생성 모달/폼
```

### UI 요구사항

| 항목 | 요구사항 |
|------|---------|
| 마감 기한 경과 표시 | 빨간 계열 색상 + 경고 아이콘으로 구분, 목록 최상단 그룹 배치 |
| 완료 처리 | 체크박스 또는 토글 버튼 1회 클릭으로 상태 변경 |
| 삭제 확인 | 실수 방지를 위한 삭제 확인 다이얼로그 제공 |
| 반응형 | 모바일(375px 이상) / 데스크톱(1280px) 브라우저 모두 정상 동작 필수 |
| 입력 유효성 | 폼 제출 전 클라이언트 측 유효성 검증 및 에러 메시지 인라인 표시 |
| 로딩 상태 | API 호출 중 로딩 인디케이터 표시 |

| AC-05 | 인증 보호 | 미인증 상태에서 할일 API 호출 시 401 반환 및 로그인 페이지 리다이렉트 | [ ] |
| AC-06 | 할일 생성 | 인증된 사용자가 제목(필수) + 마감일(필수) + 설명(선택) 입력 시 할일 생성 성공, 목록에 즉시 반영 | [ ] |
| AC-07 | 할일 생성 | 미인증 사용자가 할일 생성 시도 시 401 반환 (생성 불가) | [ ] |
| AC-08 | 할일 수정 | 인증된 사용자가 자신의 할일 제목/설명/마감일 수정 후 저장 시 즉시 목록에 반영 | [ ] |
| AC-09 | 할일 삭제 | 삭제 확인 다이얼로그 승인 후 할일 영구 삭제, 목록에서 즉시 제거 | [ ] |
| AC-10 | 완료 처리 | 1회 클릭으로 pending → completed 전환, 재클릭으로 completed → pending 되돌리기 가능 | [ ] |
| AC-11 | 마감 기한 경과 | 서버 KST 기준 오늘 날짜 > due_date AND status = pending 인 항목은 목록 최상단에 별도 그룹으로 표시 (빨간 색상/아이콘) | [ ] |
| AC-12 | 데이터 격리 | 사용자 A가 사용자 B의 할일 조회/수정/삭제 시도 시 403 반환 | [ ] |
| AC-13 | 반응형 UI | 모바일(375px 이상) 및 데스크톱(1280px) 브라우저에서 주요 화면(로그인, 목록, 생성폼) 레이아웃 깨짐 없이 정상 동작 | [ ] |

## 8. 비기능 요구사항

### 보안

| 항목 | 기준 |
|------|------|
| **비밀번호 저장** | bcrypt (cost factor 12 이상) |
| **인증 방식** | JWT (Access Token 15분 / Refresh Token 7일) |
| **토큰 저장** | Access Token: 메모리 저장 / Refresh Token: HttpOnly Cookie |
| **통신 보안** | HTTPS 적용 |
| **데이터 접근 통제** | 백엔드에서 소유자 여부 검증 필수 (타인 데이터 접근 시 403) |

### 에러 처리

**HTTP 상태 코드 기준**

| 코드 | 의미 |
|------|------|
| 200 / 201 | 성공 / 생성 성공 |
| 400 | 잘못된 요청 (유효성 검증 실패) |
| 401 | 인증 실패 (로그인 필요) |
| 403 | 인가 실패 (권한 없음) |
| 404 | 리소스 없음 |
| 500 | 서버 내부 오류 |

**표준 에러 응답 형식**

```json
{
  "status": "error",
  "code": "UNAUTHORIZED",
  "message": "로그인이 필요합니다."
}
```

## 9. 기술 스택

> 3-티어 아키텍처 (Frontend / Backend / Database) 기반으로 구성

| 계층 | 기술 | 버전 | 비고 |
|------|------|------|------|
| **Frontend** | React | 19 | UI 컴포넌트 기반 SPA |
| | TypeScript | 최신 | 타입 안전성 확보 |
| **Backend** | Node.js | LTS | 서버 런타임 |
| | Express | 최신 | REST API 서버 프레임워크 |
| | pg | 최신 | PostgreSQL 직접 연동 라이브러리 (ORM 미사용) |
| **Database** | PostgreSQL | 17 | 메인 데이터베이스 |

### 아키텍처 구조

```
┌─────────────────┐     HTTP/REST     ┌─────────────────┐     SQL     ┌─────────────────┐
│                 │ ◄────────────────► │                 │ ◄──────────► │                 │
│  React 19 + TS  │                   │ Node.js/Express  │             │  PostgreSQL 17  │
│   (Frontend)    │                   │   (Backend)      │             │   (Database)    │
└─────────────────┘                   └─────────────────┘             └─────────────────┘
```

### 기술 선택 근거

| 항목 | 결정 | 이유 |
|------|------|------|
| ORM 미사용 | `pg` 직접 사용 | SQL 제어권 확보, 불필요한 추상화 제거 |
| React 19 | 최신 버전 사용 | 최신 훅 및 성능 개선 활용 |
| PostgreSQL 17 | 도메인 정의서 DDL과 동일 | UUID, CHECK 제약, 인덱스 전략 그대로 적용 가능 |
